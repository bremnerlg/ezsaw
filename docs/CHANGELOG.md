## V1.2.0 Beta

### Overview

EZSAW V1.2.0 Beta is a fully functional PyQt5 desktop application for automotive door troubleshooting. The Beta represents a complete tool for analyzing door check measurement outliers with configurable schema, internationalization support, and comprehensive testing.

### What's Implemented

#### Core Functionality
- **VIN Entry**: Direct VIN lookup to retrieve vehicle-specific outliers
- **Make/Model/Year Dropdowns**: Filter vehicles by manufacturer, model, and production year
- **Outlier Visualization**: Scatter plots showing measurement distributions with tolerance bounds
- **Stat Family Navigation**: Sequential traversal through different test types (Closing Energy, Hinge Inclination, etc.)
- **Vehicle Context**: Each plot displays all vehicles' measurements with the current vehicle highlighted in red

#### Data Model
- **Three-table schema**: `vehicles`, `auto_door_stats`, `steps` (joint table)
- **Configurable via JSON**: All table/column names referenced through `config/db_config.json`
- **Door locations**: driver_front, driver_rear, passenger_front, passenger_rear, rear_hatch, hood
- **Outlier detection**: Measurements where result_y falls outside tolerance bounds (lower/upper limits)

#### Internationalization
- **5 languages**: English, German, French, Spanish, Dutch
- **Locale system**: JSON-driven UI strings (35 keys) plus test name translations (9 entries)
- **Runtime switching**: Language change triggers app restart with new locale
- **Per-locale DB configs**: Each language can point to a different database

#### Schema Configurability (from V3.2.0A)
- All database table and column names referenced through `config/db_config.json` keys (e.g., `EZ_STAT_NAME_FIELD`, `EZ_JOINT_TABLE_DOOR_LOCATION_FIELD`)
- The `_quote_identifier()` function ensures safe SQL generation with proper quoting
- Different deployments can use different DB schemas by swapping the config file

#### Stat Family Visualization (from V3.2.0A)
- The "next" and "previous" buttons traverse outlier results sequentially
- Each stat family is plotted as a scatter background with the current vehicle highlighted in red
- Tolerance bounds shown as horizontal lines on each graph

#### Graph Info Panel (from V3.2.0A)
- Vehicle details (VIN, make, model) displayed alongside each plot
- Test name translated from English DB values to localized display strings via `EZ_TEST_NAMES` dictionary
- Precise x/y values shown with units

  - `test_unit.py` (39 tests): Model, queries, locale, DB config
  - `test_widgets.py` (22 tests): UI state, validation, navigation, plotting
  - `test_stress.py` (47 tests): 1000-case plotting, edge cases, navigation stress
- **102 passing**, 6 failing (pre-existing Qt colour rendering issue)

### Architecture

#### File Structure

├── config/          # JSON configs (locale, DB, user prefs)
├── src/
│   ├── main.py      # PyQt5 GUI (intro_form class)

#### Key Components
- **`intro_form`**: Main QMainWindow with all UI elements
- **`test_case`**: Core data object for measurement results
- **`matricize_test_cases()`**: Converts test cases to numpy matrices for plotting
- **`_quote_identifier()`**: Safe SQL generation with proper quoting

## V3.1.1 Alpha (2026-07-17)

### Overview
UI layout overhaul following statistical software design patterns, with progressive disclosure and door-required enforcement.

### Changes

#### Layout: Left Sidebar (Statistical Software Pattern)
- Moved all controls from a horizontal controls bar to a **left sidebar** (270px fixed width, dark themed with border-radius)
- Research conducted on JMP, Minitab, SPSS, and dashboard design patterns — the sidebar pattern is the industry standard for data analysis tools because controls remain always-visible while maximizing plot area
- **Rationale:** Three-column row layout caused clipping at narrow window widths. The sidebar pattern gives:
  - Maximum vertical space for the outlier plot
  - Natural left-to-right reading flow (controls → visualization)
  - Scalability for future control additions
  - Consistent with how JMP, Minitab, and other statistical tools organize their UI
- The sidebar uses a distinctive dark-widget background (`BG_WIDGET`) with a visible border to clearly demarcate the controls panel from the plot area

#### OR Divider (Adapted for Sidebar)
- Changed from vertical "OR" divider between VIN/Vehicle columns to a **horizontal** `── OR ──` divider between VIN and Vehicle sections within the sidebar
- **Rationale:** In a vertical sidebar, a horizontal divider is the natural choice — consistent with daisyUI, Semantic UI divider patterns

#### Progressive Disclosure: Button Greying Out
- **Both** Query and Enter buttons are now **disabled (greyed out)** until a door location is selected
- New `_update_action_buttons()` method toggles `setEnabled(False/True)` on both buttons based on `_selected_door_key()` returning a valid door
- Called at initialization and on every `_on_door_changed` event
- **Rationale:** Prevents users from running queries before selecting a door (which would return empty results), matching the common "prerequisite → action" pattern in enterprise software

## V3.0.0A

### Goals and Aims with This Iteration
The major aim for 3.0.0A was (written retrospectively) to create a simple GUI with graph displays of the data found within an SQL server hosting psuedo-data based (see data/pseudo_database). The implementation was to consist of Kivy with Matplotlib integration so the out-of-tolerance results could be "walked through" in a series of steps determined by a troubleshooting flowchart.

### Conclusion

#### The Kivy Problem
After hours of attempting to get kivy to work consistently and display a layout conducive to a standard data analysis wizard, it has been concluded it is simply too bloated and clunky for use on this project. The main issues that were encountered are
    - 1. Poor documentation on layout creation and input functionality.
    - 2. A lack of native graphing support without the use of ugly backend matplotlib data structures.
    - 3. An interface not conducive to rapid testing/production, rather one that would take weeks or even months to learn all of the subtle nuances of event-handling and layouts.

#### The Proposed Solution: PyQt6
The challenge of creating a GUI is finding a decently supported and lightweight interface that will not be burdensome for one that simply wants their data analysis program to work, and complete basic functions such as displaying textbox widgets and graphs, accepting text input, and easy reproducibility across multiple devices. 

As stated by the title,  PyQt6 is the next probable lead to chase down for this aspect of the program. The attractive features of this library are: 
    - High implementation/prototyping speed.
    - Excellent interactive graphing support for PyQtGraph, potentially eliminating the need for a separate graphing library (e.g. matplotlib).
    - Excellent community support.
        - Popular and well-documented.
    - Cross-platform and easily portable.
